## [Unreleased]

### Fixed
- fix(scrapes): recurring user searches die permanently on worker restart (#203). User recurring searches self-reschedule via `scrape_kleinanzeigen.apply_async(countdown=interval)` — a pending ETA task that lives only in the broker. A worker restart/crash (or any run that errors before the reschedule line) drops it and the chain stops forever, with no recovery — unlike admin searches, which Celery Beat re-dispatches from `next_run_at` every 60s. Live impact: search 305 (5-min interval) sat 49 min stale after the worker was rebuilt for #199/#201. Added a `scrape.reap_stale_recurring_searches` beat task (every 120s) that re-primes any recurring `scrape_task` whose `last_run_at` is older than `interval + grace` (grace = max(interval/2, 120s), so a healthy or in-flight chain is never raced). Re-priming atomically bumps `last_run_at` (same claim trick as `dispatch_admin_searches`) so overlapping passes can't double-dispatch; `running`/`pending`/`cancelled` and one-shot (no `interval_seconds`) tasks are left alone. Added `tests/test_recurring_reaper.py` (6 cases: revive-overdue, idempotent-second-pass, healthy-left-alone, one-shot-skipped, running/pending/cancelled-ignored). Verified: full suite 56 passed, ruff clean on `app/`; live search 305 revived and re-scheduling at 5-min cadence.
- fix(notifications): notification gating fixes #185–#189 (one branch). (1) Remove the "deals-only" feature entirely (#185) — its toggle gated notifications on a Core/Pro-gated deal badge, so a Basic user who flipped it received nothing; removed the UI, the `deals_only` payload field, both worker guards, the `User.deals_only_enabled` model column, and dropped it via migration `0030`. (2) Push toggle now actually fires push (#186): the toggle's change handler POSTs `push_enabled` to `/api/settings/notifications` on subscribe/unsubscribe so `User.push_notifications_enabled` is set in lockstep with the browser subscription. (3) `push_enabled` is now optional in `NotificationSettingsPayload` (`bool | None = None`) and `update_notification_settings` only touches the DB flag when present (#187), so saving email/other settings no longer force-enables push. (4) The scrape task now passes `task_id=resolved_task_id` to `_send_email_notifications` (#188) so every email `NotificationDelivery` row carries the correct task_id for per-search auditing. (5) Added `tests/test_notification_gating.py` (#189) covering email/push gating, the settings POST no-toggle-push contract, and delivery-row task_id/status. Verified: full suite 49 passed.
- fix(notifications): make test pushes visibly pop (#193). Both test paths (`/push/test` and the admin `/api/admin/test-notification` via `run_test_push`) reused `tag="search-0"`, so every repeat click silently **replaced** the previous OS toast instead of creating a fresh one — the red +1 badge still bumped (it's driven by the service worker's `notificationShown` message, not by whether the user saw a popup), which made it look like "no notification fired." `_send_push_notifications` now takes optional `tag`/`title` overrides; tests pass a per-click millisecond timestamp tag and a `"TEST - kleeblatt.space"` title so each click is a distinct, visible notification. Real notifications keep grouping by `search-{task_id}`. Verified: full suite 49 passed.
- fix(notifications): prune expired push subs + fix test task_id FK (#194). (1) The 404/410 "subscription expired" prune was **dead code**: the retry wrapper (`_retry_with_backoff`) swallowed the `WebPushException` and returned it as `last`, so the outer `except WebPushException` (which appended to `stale` and deleted) never ran — expired endpoints lingered and failed every future test silently (live repro: user 10 returned `removed:0` on a `410 Gone`). Now the loop inspects `last.response.status_code` directly and prunes on 404/410. (2) Both test paths passed `task_id=0`, which doesn't exist in `scrape_tasks`, so `_save_notification_delivery` hit a FK violation and aborted the request — every "send test notification" click crashed on delivery logging. Tests now pass `task_id=None` (the column is nullable). Added `test_expired_subscription_is_pruned_on_410`. Verified: full suite 50 passed; live re-probe shows user 10 `removed:1`, user 3 `sent:1`, no FK error.
- fix(ci): drop unused `WebPushException` import breaking ruff (#197). The #194 refactor switched the push loop to inspect `last.response.status_code` instead of catching `WebPushException`, leaving the name imported but unused. Ruff (CI gate) flagged `F401` and failed every "CI & Deploy" run since #194 merged. Removed the unused import. Verified: `ruff check` clean, full suite 50 passed.
- fix(auth): Google OAuth callback fails with mismatching_state (#199). The docker-compose api `command` dropped uvicorn's `--proxy-headers --forwarded-allow-ips "*"` (present in the Dockerfile CMD), so behind the Caddy TLS proxy uvicorn ignored `X-Forwarded-Proto` and always built an `http://` redirect_uri via `request.url_for()`. The OAuth state cookie got mis-scoped, so the callback raised `MismatchingStateError` and every Google login died with "Google sign-in failed." Fixed two ways: (1) `google_login` now forces the redirect_uri to `https://` so it never depends on scheme detection; (2) restored the proxy-header flags in docker-compose.yml. Also added `logger.exception` in the callback so future OAuth failures are diagnosable instead of swallowed. Verified live: state round-trip now matches (error changed from `mismatching_state` to `invalid_grant` on a dummy code, i.e. exchange proceeds); `ruff check` clean; full suite 50 passed.
- fix(csrf): www host 403s every POST incl. test-notification button (#201). Caddy served both `kleeblatt.space` and `www.kleeblatt.space` but proxied `Host: kleeblatt.space` upstream, so a visitor on the `www.` origin sent `Origin: https://www.kleeblatt.space` and the `security_middleware` CSRF check (host != request host) returned 403 on every POST — login/settings saves AND the in-app test-notification buttons failed silently, while server-initiated pushes (no Origin) and GET/OAuth flows worked. Fixed two ways: (1) Caddy now redirects `www.` -> apex (canonical host); (2) the middleware strips a leading `www.` from both hosts before comparing, so same-site is never mistaken for cross-origin. Verified live: `www.` Origin POST now returns `sent:1` (was 403); `www.kleeblatt.space` -> 301 to apex; apex POST still 200. `ruff` clean; suite 50 passed.
### Removed
- fix(notifications): remove the quiet-hours feature entirely (#184). Quiet hours silently suppressed **all** email and push notifications: the settings page hardcoded the time inputs to `22:00`/`08:00` even when unset and had no enable toggle, so `saveNotificationSettings()` wrote a phantom quiet window the moment a user touched any notification setting; the worker then early-returned for ~10h/day. It also compared against UTC instead of the user's local time (1–2h off for DE users). Removed the quiet-hours UI + `.quiet-hours` CSS, the `_in_quiet_hours()` helper and both call-site guards in `app/worker/tasks.py`, the `quiet_start`/`quiet_end` fields from `NotificationSettingsPayload`/`update_notification_settings`/settings_page/GDPR export, and dropped the `users.quiet_start`/`quiet_end` columns via migration `0029`. Verified: full suite 41 passed; alembic history linear (single head `0029`).

### Fixed
- fix(dashboard): populate location-id hidden field. The location autocomplete JS referenced the old variable `locationIdInput` after it was renamed to `locIdField`, so the hidden location-id input was never populated — location-based searches silently lost their location filter. Restored the correct reference.
- fix(ci): make the deploy step resilient to a dirty VPS working tree. The SSH deploy does `git checkout main`; an uncommitted server-side edit to `dashboard.html` aborted it and failed the #151 deploy run. Added `git checkout -- .` before `git checkout main` so the VPS tree always reflects the committed SHA (real changes belong in a PR). The #151 deploy was re-run successfully after clearing the stray edit.
- fix(auth): stop 500 on successful registration from non-ASCII flash cookies (#149). The `register` handler set `flash_success`/`flash_error` cookies with German copy containing `–`/`ü`/`ä`; Starlette encodes Set-Cookie as latin-1, so the response raised `UnicodeEncodeError` and turned every successful signup (SMTP configured) into a 500. Added `app/shared/cookies.py:ascii_cookie()` (transliterates umlauts, maps dashes, drops unmappable bytes) and routed every user-facing flash cookie through it in `auth.py`, `billing.py` and `scrapes.py`. Added `tests/test_flash_cookie_encoding.py`. Verified: registration now 303 + logged-in, full suite 24 passed.
- fix(migrations): resolve Alembic multi-head fork (#124). The slug chain `52ad6a85a96f` -> `a43ba04bf415` was stranded as a second head so `alembic upgrade heads` (run by `deploy/deploy.sh`) aborted and broke deploys. Added merge migration `5f17314ef463` joining both heads, and made `a43ba04bf415` idempotent (guards every drop/index/column) so re-running migrations is safe against a partially-applied DB.
- fix(security): remove hardcoded default admin password `KaSearch2026` from `config.py` (#125). `APP_PASSWORD` now has no built-in default; the bootstrap admin path stays disabled unless a strong password is supplied via env. The validator also rejects the old leaked value (public in git history) and any empty password when `BOOTSTRAP_ADMIN_ENABLED=true` outside dev. Rotate the credential on any live deployment.
- fix(security): back the rate limiter with Redis (#128). The slowapi `Limiter` used the default in-memory storage, so each API replica kept its own counters — an attacker got N auth attempts *per process*, defeating the brute-force throttle in multi-replica deploys. It now uses `storage_uri=settings.redis_url` for a shared window, with `in_memory_fallback_enabled=True` so a brief Redis outage degrades to per-process limiting instead of 500-ing every request. Storage init is lazy, so the CI import-smoke still runs without Redis.

### Fixed
- fix(ci): resolve `ModuleNotFoundError: No module named 'app'` failing the CI `test` job on every run. The `test` job invokes `pytest -q` as a console script, which does not add the repo root to `sys.path` (unlike `python -m pytest`, which is why it passed locally). `tests/conftest.py` does `import app.api.main`, so the job aborted at conftest import. Added `pythonpath = .` to `pytest.ini` so the repo root is always on the path regardless of launcher. Verified: `pytest -q` now runs 21 passed (exit 0).
