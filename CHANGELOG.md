### Fixed
- fix(email): diagnose + harden Resend email sending for #237. The new-results email job (`scrape.kleinanzeigen` → `_send_email_notifications`) was failing for every recipient with `401 validation_error → "API key is invalid"` (Sentry `PYTHON-FASTAPI-1C`, 24 hits, live since 2026-07-11 ~06:00 UTC). Root cause confirmed from *inside the running worker* (read-only `GET /domains` against Resend): the deployed `RESEND_API_KEY` is **`suspended_api_key`** — i.e. revoked/expired at Resend, not a code bug. Two deploy-side gaps made recovery fragile and would bite even after rotation: (1) `RESEND_API_KEY` and `EMAIL_FROM` were never injected through `docker-compose.yml`'s `environment:` blocks for the `api`/`worker` services, so they were only picked up via pydantic-settings' `.env` read at import time — a `.env` edit could not take effect without a full process restart. Both are now passed through explicitly (`${RESEND_API_KEY:-}` / `${EMAIL_FROM:-onboarding@resend.dev}`) with comments noting rotation = edit `.env` + `docker compose up -d`. (2) `email_from` defaulted to the Resend sandbox sender `onboarding@resend.dev`, which can only deliver to the account owner's inbox — set a Resend-verified domain via `EMAIL_FROM` for external recipients. Also made the send failure self-diagnosing: the non-2xx branch now parses Resend's JSON error `name`/`message` (e.g. `suspended_api_key`, `invalid_api_key`) into the log/Sentry message instead of dumping a raw body. The suspended key itself must be rotated at resend.com (no Resend credentials available in-repo); once a valid key is in `.env` and the containers are reloaded, #237 closes. Verified: `docker compose config -q` clean, `py_compile app/shared/email_notifications.py` OK, `tests/test_notification_gating.py` 9/9 still pass.
- fix(email): during the temporary Resend suspension (#237), surface a non-blocking in-app banner instead of silently dropping mail. New `app/shared/email_status.py` tracks a process-shared degraded marker (mtime-pinned file, 6h window) that the worker sets on every failed new-results send (`_send_email_notifications` in `app/worker/tasks.py`, via `mark_email_failed()`) and clears on a successful send (`clear_email_failed()`); `app/api/emailer.py` sets the same marker when a verification email is rejected. `app/api/version.py:register_globals` now exposes `email_degraded()` / `email_degraded_since()` to Jinja, and a new `app/api/templates/email_status_banner.html` include renders the German warning "⚠️ E-Mail-Versand vorübergehend nicht verfügbar. Verifizierungs- und Benachrichtigungs-Mails können aktuell nicht zugestellt werden…" — included on dashboard, settings and pricing. The banner evaluates the *function* per request (registered as a callable, called with `()` in the template) so it self-heals the moment sends succeed again. Delivery rows are still written as `failed` for auditing. Verified: `py_compile` clean, `tests/test_notification_gating.py` 9/9 pass, Jinja render confirmed to show only when degraded (0 bytes when healthy).

### Added

New `app/shared/result_filters.py` (`parse_terms` + `passes_filters`) implements the locked semantics: require = ALL terms (AND), exclude = NONE, both matched as WHOLE WORDS case-insensitively across title + description; exclude_locations = substring against the result's location (so "Saarbrücken" or the PLZ "66111" both catch "66111 Saarbrücken"). - fix(search): category filter never actually scoped the search — a "Häuser zum Kauf" or "Dienstleistungen" search returned listings from every category (jobs, electronics, …), which is why results didn't match the selected category. Root cause in `app/shared/url_builder.py`: kleinanzeigen scopes a category via a numeric `c{id}` code packed into the URL's `k` token (e.g. `…/k0c208`), exactly like `l{locationId}` for location — the `s-{category}` path segment is display/SEO text only. The builder only ever emitted the slug as that path segment with a bare `k0` token, and `k0` means "all categories", so every category search silently degraded to a full-text search for the slug word across the whole marketplace. There was no slug→id mapping anywhere in the repo, so category scoping had in fact never worked. - fix(scrapes): recurring user searches die permanently on worker restart (#203). User recurring searches self-reschedule via `scrape_kleinanzeigen.apply_async(countdown=interval)` — a pending ETA task that lives only in the broker. A worker restart/crash (or any run that errors before the reschedule line) drops it and the chain stops forever, with no recovery — unlike admin searches, which Celery Beat re-dispatches from `next_run_at` every 60s. Live impact: search 305 (5-min interval) sat 49 min stale after the worker was rebuilt for #199/#201. Added a `scrape.reap_stale_recurring_searches` beat task (every 120s) that re-primes any recurring `scrape_task` whose `last_run_at` is older than `interval + grace` (grace = max(interval/2, 120s), so a healthy or in-flight chain is never raced). Re-priming atomically bumps `last_run_at` (same claim trick as `dispatch_admin_searches`) so overlapping passes can't double-dispatch; `running`/`pending`/`cancelled` and one-shot (no `interval_seconds`) tasks are left alone. Added `tests/test_recurring_reaper.py` (6 cases: revive-overdue, idempotent-second-pass, healthy-left-alone, one-shot-skipped, running/pending/cancelled-ignored). Verified: full suite 56 passed, ruff clean on `app/`; live search 305 revived and re-scheduling at 5-min cadence.

- fix(auth): Google OAuth callback fails with mismatching_state (#199). The docker-compose api `command` dropped uvicorn's `--proxy-headers --forwarded-allow-ips "*"` (present in the Dockerfile CMD), so behind the Caddy TLS proxy uvicorn ignored `X-Forwarded-Proto` and always built an `http://` redirect_uri via `request.url_for()`.
